# Dashiai Narrative Shapes — 叙事弧分类

## 核心原则

每张 PPT 必须服务于一个**清晰的论点**，而非罗列话题。Strategist 在 Confirmation c 中选择叙事弧，决定整份演示的逻辑流。

## 7 种叙事弧

### 1. problem-to-solution

- **结构**：现状痛点 → 根因分析 →  proposed 方案 → 预期收益
- **适用**：技术选型、流程优化、变革管理提案
- **关键句**："如果我们不解决这个问题，______ 将会发生；如果我们这样做，______ 会改善。"
- **证据需求**：痛点的量化数据、方案的可行性证明

### 2. strategic-recommendation

- **结构**：SCQA（情境→冲突→问题→答案）→ 战略选项 → 推荐方案 → 实施路径
- **适用**：高管汇报、战略方向决策、资源分配
- **关键句**："基于 ______ 分析，我们推荐 ______，因为它在 ______ 维度上最优。"
- **证据需求**：市场分析、竞品对标、财务模型

### 3. investment-case

- **结构**：投资机会 → 市场规模 → 竞争优势 → 财务预测 → 风险与缓解
- **适用**：融资路演、投资决策、BP
- **关键句**："这是一个 ______ 亿的市场机会，我们拥有 ______ 壁垒，寻求 ______ 轮融资。"
- **证据需求**：TAM/SAM/SOM、单位经济模型、term sheet

### 4. operating-review

- **结构**：周期概览 → 核心指标达成 → 业务线详情 → 问题与改进 → 下期展望
- **适用**：季度经营分析、年度总结、董事会汇报
- **关键句**："本周期我们实现了 ______，超出/低于预期 ______，主要原因是 ______。"
- **证据需求**：KPI 数据、同比/环比、预算对比

### 5. proposal

- **结构**：背景 → 需求理解 → 解决方案 → 团队与里程碑 → 报价
- **适用**：投标、客户提案、服务范围确认
- **关键句**："理解您的需求 ______，我们的方案通过 ______ 来满足，总投入 ______。"
- **证据需求**：案例研究、团队资质、详细 WBS

### 6. launch-story

- **结构**：用户洞察 → 产品发布 → 核心价值 → 市场反馈 → 下一步
- **适用**：新产品发布、营销 campaign、产品迭代
- **关键句**："我们发现用户需要 ______，所以我们创造了 ______，市场反馈 ______。"
- **证据需求**：用户调研、早期采用数据、媒体曝光

### 7. teaching-narrative

- **结构**：学习目标 → 概念讲解 → 示例演示 → 练习/案例 → 总结
- **适用**：培训材料、知识转移、教学课件
- **关键句**："学习结束后，你将能够 ______。让我们通过 ______ 来理解这个概念。"
- **证据需求**：学习成果度量、练习题答案、参考资料

## 叙事弧选择决策树

```
用户需要做决策？
  ├── 是的，选择方案/方向 → strategic-recommendation
  ├── 是的，投入资金 → investment-case
  ├── 是的，批准项目 → proposal
  └── 不，主要是传达信息/教育
      ├── 需要改变对方观点 → problem-to-solution
      ├── 回顾过去周期表现 → operating-review
      ├── 发布新事物 → launch-story
      └── 传授知识 → teaching-narrative
```

### 重叠场景的 tie-breaker

七种形状是“主叙事选择”，不是互斥的内容标签。一份 deck 可以包含提案、问题和实施内容，但 `narrative_shape` 只能记录一个主弧。发生重叠时按首要决策排序：

1. 是否要求投入资金或融资判断？是 → `investment-case`。
2. 是否要求客户/管理层批准一个已定义项目或服务范围？是 → `proposal`。
3. 是否要求在多个方向中选择并采纳推荐？是 → `strategic-recommendation`。
4. 是否以复盘一个周期的表现为主要目的？是 → `operating-review`。
5. 是否以发布新产品、能力或组织变化为主要目的？是 → `launch-story`。
6. 是否以可测量的学习结果为主要目的？是 → `teaching-narrative`。
7. 其余需要先建立问题紧迫性再推动改变的场景 → `problem-to-solution`。

若首要决策仍不清楚，Strategist 必须在八项确认中询问，不能根据页面数量猜测。

## 论点（Thesis）撰写规范

- **一句话**，不超过 25 个字（中文）或 20 个词（英文）
- 必须是**主张**（claim），不是话题（topic）
  - 好：「微服务架构将部署效率提升 300%」
  - 坏：「微服务架构介绍」
- 必须是**可争论**的——有人可能不同意

## 证据缺口（Evidence Gaps）

Strategist 必须标注每页的证据状态：

| 状态 | 含义 | 处理方式 |
|------|------|---------|
| verified | 有数据/事实支撑 | 正常展示 |
| assumption | 合理推断但无直接数据 | 标注为假设，建议验证 |
| missing | 需要补充的证据 | 在 speaker notes 中标注风险 |

## 对 Executor 的影响

叙事弧决定：
- **page_rhythm**：problem-to-solution 通常在"根因分析"后设置 breathing 页
- **layout 选择**：investment-case 偏好 big-number + chart；teaching-narrative 偏好步骤化布局
- **取回标题（Takeaway）位置**：strategic-recommendation 必须在页标题中体现结论
